# 项目原则

## 1. 安全优先

工具默认只读。删除、移动、隔离、注册表修改、服务修改都属于高风险操作，必须有明确证据、测试覆盖和可恢复方案。

禁止把“不认识的文件”等同于“可以删除的文件”。

## 2. 解释优先

每个候选文件或目录都必须尽可能给出解释：

- 可能的所属应用或系统组件
- 发现依据
- 最近访问、修改、创建时间
- 当前是否被进程占用
- 是否出现在服务、计划任务、启动项、PATH、快捷方式、注册表或安装记录中
- 删除、隔离、保留的建议理由

如果解释不足，结论必须降低置信度。

## 3. 证据驱动

每个风险判断都必须能追溯到证据，而不是只输出一个分数。

示例证据：

- Windows Installer 或卸载注册表记录
- Microsoft Store package 信息
- 运行中进程的模块列表
- 服务可执行文件路径
- 计划任务 action 路径
- 启动项、快捷方式、文件关联、COM 注册
- 文件签名和发布者
- 已知安全清理规则

## 4. 保守删除

工具应优先建议：

1. 保留
2. 交给 Windows 官方工具清理
3. 隔离并保留恢复点
4. 删除

删除永远是最后一步。

## 5. 系统边界不可突破

以下位置默认禁止直接删除或需要极强警告：

- `C:\Windows\Installer`
- `C:\Windows\WinSxS`
- `C:\Windows\System32`
- `C:\Windows\SysWOW64`
- `C:\Windows\SystemApps`
- `C:\Windows\servicing`
- `C:\Windows\INF`
- `C:\Windows\System32\DriverStore`
- 程序安装目录内的 DLL、EXE、配置和插件目录

能由 Windows 官方工具处理的系统项目，优先调用或提示官方工具，而不是手动删除文件。

## 6. 可恢复

所有非临时文件的清理都应优先进入隔离区，并记录恢复元数据：

- 原始路径
- 文件哈希
- 文件大小
- 发现证据
- 风险等级
- 操作时间
- 操作版本

## 7. TDD 优先

核心逻辑必须遵守 Red-Green-Refactor：

1. 先写失败测试
2. 写最小实现通过测试
3. 重构并保持测试通过

文件归属识别、风险评分、清理计划生成、禁止目录保护、隔离恢复都必须先有测试。

## 8. Agent 可维护

项目文档必须让后续 AI Agent 能快速接手：

- 每次重要决策写 ADR
- 每次任务更新 `PROGRESS.md`
- 不把隐含规则只放在对话里
- 任务必须可验证
- 不做无关重构

## 9. 子 Agent 分工优先

对于可以拆分的单个任务，应尽可能使用子 Agent 并行处理清晰、独立、可验证的子任务，减少主控对话 context 消耗。

适合拆分的任务：

- 文档补充和审查
- 独立模块探索
- 测试用例补充
- 互不重叠文件范围内的实现
- CI、依赖、配置等旁路验证

不适合拆分的任务：

- 当前主流程的直接阻塞点
- 文件写入范围高度重叠的修改
- 需要连续细粒度设计判断的核心安全逻辑
- 需要立即根据本地运行结果反复调整的变更

主控 Agent 仍需负责最终集成、验证和进度记录。
